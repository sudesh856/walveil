SECURITY
Critical Security Considerations
These are not afterthoughts. Each item below maps to a specific implementation point in the phases. Read this section first — it informs every architectural decision.
SEC-01 — Replication Credentials Never in Config Plaintext

Never store replication_password in plaintext YAML/TOML. At minimum: support reading from environment variables with $ENV_VAR interpolation. Preferably: support reading from files (Docker secrets, Kubernetes secrets). Never log credentials. Mask them in debug output.

In Phase 0, design the config struct with a SecretRef type that abstracts the source. In Phase 4, add a secret resolution layer that supports env, file, and Vault/SOPS.

SEC-02 — Replication Slot Poisoning

PostgreSQL replication slots accumulate WAL until a consumer acknowledges LSN progress. If walveil crashes or stalls without advancing the LSN, the Postgres server will grow WAL on disk indefinitely — this is a silent disk exhaustion attack vector. A malicious or misconfigured consumer can OOM the database host.

Mitigation: implement a slot warden goroutine that monitors lag (via pg_replication_slots.confirmed_flush_lsn vs pg_current_wal_lsn()). If lag exceeds a configurable threshold (default: 1GB or 5min), emit a critical alert and optionally drop the slot to protect the database. This is implemented in Phase 3.

SEC-03 — Webhook SSRF (Server-Side Request Forgery)

Walveil will make outbound HTTP calls to user-configured webhook URLs. A misconfigured or malicious URL could point to internal services, cloud metadata endpoints (169.254.169.254), or loopback. Walveil must validate webhook targets before connecting.

Mitigation: resolve the hostname at startup, reject RFC 1918 and link-local addresses (10.x, 172.16-31.x, 192.168.x, 127.x, 169.254.x, ::1). Also block 0.0.0.0 and [::]. Re-validate on DNS resolution changes. Implement in Phase 2.

SEC-04 — TLS Enforcement for All Connections

PostgreSQL supports TLS. Your replication connection must verify the server certificate. Default sslmode=verify-full. Allow downgrade only via explicit config flag with a startup warning. All outbound webhook connections must use TLS. HTTP (non-TLS) webhook targets should require an explicit opt-in flag.

SEC-05 — Webhook Payload Signing (HMAC)

Every outbound webhook payload must include an HMAC-SHA256 signature in the X-Walveil-Signature header. Key is configured per-target. Receivers can verify authenticity. This prevents spoofed replays even if the webhook endpoint is public. Implement in Phase 2.

SEC-06 — Decoded Data Leakage via Logs

Decoded row data (INSERT/UPDATE/DELETE values) often contains PII: emails, passwords (even hashed), API keys. Walveil must never log row values at INFO or below. Debug logging of row data must be gated behind a separate --log-level=trace flag, and every trace log of row values must include a visible warning prefix.

Additionally: support a column-level redaction list in config so specific columns are masked with [REDACTED] in both logs and delivery payloads if needed.

SEC-07 — LSN State File Integrity

The file walveil writes to persist LSN checkpoints is a critical piece of state. If it's corrupted or tampered with, walveil may re-deliver old events or skip events entirely. Protect this file: write atomically (write to .tmp, then rename()), include a checksum (CRC32 or SHA256), and on startup, verify the checksum before trusting the value. Implement in Phase 1.

SEC-08 — Rate Limiting on Webhook Delivery

During a bulk import or schema change, walveil may decode thousands of events per second. Sending them all to a webhook target at full speed will either overwhelm the target or cause them to block walveil's delivery loop. Implement configurable per-target rate limiting using a token bucket. Also implement back-pressure: if the delivery queue fills above 80%, slow WAL consumption to avoid unbounded memory growth.

SEC-09 — Replication User Principle of Least Privilege

The database user walveil connects as should have only what is required: REPLICATION privilege + SELECT on target tables (needed for initial snapshot if implemented). It must NOT be a superuser. Your documentation and startup checks must verify this and warn loudly if the connected user has superuser rights.

SEC-10 — Config Validation at Startup (Fail Fast)

All configuration must be validated before any connection is opened, any slot is created, or any goroutine is spawned. Unknown keys should be rejected (prevents silent misconfiguration). Required fields missing should exit with a clear error. Invalid URLs, invalid DSNs, invalid table names — all must be caught at startup, not at runtime after state has been modified.

SEC-11 — Replication Slot Naming Collision

If two walveil instances are configured with the same slot name against the same Postgres instance, they will both try to consume from the same slot — leading to split delivery and LSN confusion. Walveil should generate a deterministic slot name from the config hash, and detect slot ownership conflicts at startup using an advisory lock.

SEC-12 — Memory Safety Under High Throughput

Decoded WAL messages are buffered in memory before delivery. A table with very wide rows or a burst of writes can cause memory spikes. Implement a max buffer size (in bytes, not events). If the buffer is full, apply back-pressure by pausing XLogData consumption. Expose buffer utilization as a metric.

PATCH: Startup Validation Before Any Side Effects

Non-negotiable. Validate every field in your config before you open a single connection, create a replication slot, or start a goroutine. The dangerous failure mode is partial initialization: walveil creates the replication slot successfully, then hits an invalid webhook URL and crashes — leaving an orphaned slot on Postgres that is now silently accumulating WAL with nothing consuming it.

Your validation pass must cover all of the following before touching any external state:

V1
DSN format validation
Parse the DSN string before attempting any connection. If it fails to parse, exit immediately with a clear error message that identifies the field. Never let a malformed DSN reach pgconn.Connect().
V2
URL format + SSRF pre-check
For every configured webhook URL: validate the URL format, then resolve the hostname and run the full SSRF blocklist check (RFC 1918, loopback, link-local, 0.0.0.0, ::1, fc00::/7, fe80::/10). Fail loudly on any violation — before the slot is created.
V3
Required fields
Every required field must be present and non-empty. Emit a specific error per missing field — not a generic "config invalid". Include the field path (e.g. source.slot_name) in the error message.
V4
Unknown keys rejected
TOML strict-decode with a struct tag that rejects unknown keys. A typo like ssl_mmode instead of ssl_mode should be a fatal error, not a silently ignored key that leaves TLS disabled.
V5
File paths
For every configured file path (cert files, checkpoint path, secret files): verify the path is syntactically valid and that the parent directory exists and is writable. A checkpoint path pointing at a read-only directory is a startup error, not a runtime surprise.
// internal/config/config.go func (c *Config) Validate() error { // Called ONCE before any external state is touched. // Returns an error listing ALL violations at once (not just the first). var errs []string if _, err := pgconn.ParseConfig(c.Source.DSNValue()); err != nil { errs = append(errs, fmt.Sprintf("source.dsn: %v", err)) } for i, sink := range c.Sinks { if sink.Type == "webhook" { if err := validateWebhookURL(sink.URL); err != nil { errs = append(errs, fmt.Sprintf("sinks[%d].url: %v", i, err)) } } } // ... all other checks ... if len(errs) > 0 { return fmt.Errorf("config validation failed:\n - %s", strings.Join(errs, "\n - ")) } return nil }

PATCH 7: Secrets Handling — No Accidental Leaks

Your DSN contains the database password. Your webhook config contains HMAC signing keys. These values flow through your application at startup and live in your config struct for the lifetime of the process. The leak paths are numerous: a logger that serializes the config struct, an error message that includes the raw DSN on parse failure, a debug endpoint that dumps internal state, a panic stack trace.

The mitigation has two parts. First, load secrets only from environment variables or files — never hardcoded in config. Second, implement a SecretRef type that wraps secret values and whose String() and MarshalJSON() methods return [REDACTED], so the value cannot accidentally serialize into any output regardless of how it gets there. Sanitize at the point you construct values, not at the point you log them.

// internal/config/secrets.go type SecretRef struct { value string } func (s SecretRef) String() string { return "[REDACTED]" } func (s SecretRef) MarshalJSON() ([]byte, error) { return []byte(`"[REDACTED]"`), nil } func (s SecretRef) Value() string { return s.value } func NewSecretRef(raw string) SecretRef { return SecretRef{value: raw} } // Resolver: env var interpolation ($VAR) and file:// paths func ResolveSecret(ref string) (SecretRef, error) { if strings.HasPrefix(ref, "$") { val := os.Getenv(ref[1:]) if val == "" { return SecretRef{}, fmt.Errorf("env var %s not set", ref[1:]) } return NewSecretRef(val), nil } if strings.HasPrefix(ref, "file://") { b, err := os.ReadFile(ref[7:]) if err != nil { return SecretRef{}, err } return NewSecretRef(strings.TrimSpace(string(b))), nil } return NewSecretRef(ref), nil // literal fallback (dev only) }

PHASE 1
WAL Core Engine
The heart of walveil. Implement the PostgreSQL logical replication protocol from scratch. This is the hardest phase technically — get it right before building anything on top of it.
How PostgreSQL Logical Replication Actually Works
01
Replication Connection
Connect with replication=database parameter. Use pgconn.Connect(ctx, dsn) directly — not a pool.
02
Identify System
Send IDENTIFY_SYSTEM. Store the system identifier for cross-cluster detection on every reconnect.
03
Create / Validate Replication Slot
Send CREATE_REPLICATION_SLOT walveil_main LOGICAL pgoutput NOEXPORT_SNAPSHOT. If slot exists, query pg_replication_slots for confirmed flush LSN.

Start Replication Stream
Send START_REPLICATION SLOT walveil_main LOGICAL lsn (proto_version '2', publication_names 'walveil_pub'). Connection becomes server-push only except for keepalive responses.
05
Message Loop
Read XLogData and PrimaryKeepalive messages. Decode pgoutput binary payload on each XLogData.
06
Standby Status Update
Periodically send StandbyStatusUpdate to acknowledge LSN progress. Only acknowledge LSNs for events you have confirmed delivered.
PATCH 1: LSN Ack Correctness — No Data Loss

Postgres tracks how far you've consumed WAL via the LSN you acknowledge back. The rule is simple: only acknowledge an LSN after every configured sink has confirmed it received and accepted that event. If you ack early and crash before the webhook fires, Postgres has already cleaned up that WAL segment — those events are gone with no way to recover them.

Acknowledging late just means you re-deliver on restart, which is acceptable. Your checkpoint file and your standby status update to Postgres must be gated on the same moment: "all sinks said yes."

// stream.go — the ack must flow AFTER dispatcher confirms delivery func (s *Stream) processXLogData(msg pglogrepl.XLogData) error { event, err := s.decoder.Decode(msg.WALData) if err != nil { return err } // Block until ALL sinks confirm. Only then advance the confirmed LSN. if err := s.dispatcher.DispatchAndWait(event); err != nil { return err // do NOT advance LSN on delivery failure } s.confirmedLSN = msg.WALStart // advance only after confirmed delivery return nil } // Keepalive flush: send StandbyStatusUpdate with confirmed LSN, not current WAL position func (s *Stream) sendStandbyStatus() error { return pglogrepl.SendStandbyStatusUpdate(s.ctx, s.conn, pglogrepl.StandbyStatusUpdate{ WALWritePosition: s.confirmedLSN, WALFlushPosition: s.confirmedLSN, WALApplyPosition: s.confirmedLSN, ClientTime: time.Now(), ReplyRequested: false, }) }
PATCH 4: Transaction Boundaries — No Logical Corruption

WAL arrives as a stream of individual row changes, but those changes belong to transactions. A transaction might insert into orders, then order_items, then update inventory — and all three only matter together. If you emit events as you decode them, you'll fire the orders insert before the transaction commits, meaning downstream systems may act on data that gets rolled back.

You must buffer all changes belonging to a transaction in memory, keyed by XID, and only emit them as a batch when you see the COMMIT message. The Begin message opens the buffer, every DML message appends to it, and Commit flushes it downstream as an atomic unit.

// internal/replication/decoder.go type txnBuffer struct { xid uint32 commitLSN pglogrepl.LSN events []*wal.ChangeEvent } type Decoder struct { relations *wal.RelationCache txns map[uint32]*txnBuffer // keyed by XID } func (d *Decoder) Decode(data []byte) ([]*wal.ChangeEvent, error) { msgType := data[0] switch msgType { case 'B': // Begin — open buffer for this XID msg, _ := pglogrepl.ParseBeginMessage(data[1:]) d.txns[msg.Xid] = &txnBuffer{xid: msg.Xid} case 'I', 'U', 'D': // DML — append to buffer, do NOT emit yet event, xid, _ := d.decodeDML(msgType, data[1:]) if buf, ok := d.txns[xid]; ok { buf.events = append(buf.events, event) } return nil, nil // nothing emitted until commit case 'C': // Commit — flush the batch atomically msg, _ := pglogrepl.ParseCommitMessage(data[1:]) buf := d.txns[msg.TransactionID] events := buf.events delete(d.txns, msg.TransactionID) return events, nil // downstream sees the full committed txn at once } return nil, nil }
PATCH 5: Reconnect & System ID Verification — No Cross-Cluster Corruption

When walveil loses its replication connection and reconnects, the naive implementation just reconnects and resumes from the last checkpointed LSN. The problem: in a failover scenario, you may reconnect to a different Postgres instance — a promoted replica, or a different host entirely — whose WAL history diverges from where you left off. LSNs are not globally unique across clusters; the same LSN value means different things on different instances.

On every reconnect, send IDENTIFY_SYSTEM and compare the returned system identifier against what you stored on first connect. A mismatch means you are talking to a different cluster. Hard fail immediately. Log clearly. Require human intervention. Do not attempt to continue.

// internal/replication/conn.go func (c *Conn) reconnect(ctx context.Context) error { backoff := newExponentialBackoff(1*time.Second, 60*time.Second) for { conn, err := pgconn.Connect(ctx, c.dsn.Value()) if err != nil { c.log.Warn("reconnect failed, backing off", zap.Error(err)) select { case <-time.After(backoff.Next()): case <-ctx.Done(): return ctx.Err() } continue } // CRITICAL: verify system ID on every reconnect sysID, err := pglogrepl.IdentifySystem(ctx, conn) if err != nil { conn.Close(ctx); continue } if c.storedSystemID != "" && sysID.SystemID != c.storedSystemID { // HARD FAIL — different cluster. Do not resume. return fmt.Errorf( "FATAL: system ID mismatch: expected %s, got %s — possible failover to different cluster, manual intervention required", c.storedSystemID, sysID.SystemID, ) } c.storedSystemID = sysID.SystemID c.conn = conn backoff.Reset() return nil } }

PATCH 2: Backpressure — No Memory Death

Your WAL decoder can read faster than your sinks can deliver. Without backpressure, decoded events pile up in memory until the process gets OOM-killed. The fix is a bounded queue between the decoder and the dispatcher. When that queue is full, you stop reading from the Postgres replication stream entirely — you don't buffer more, you don't drop events, you just pause.

Postgres is fine with this; it holds the WAL and waits. The pressure travels back to the database instead of exploding your heap.

// internal/delivery/dispatcher.go type Dispatcher struct { sinks []Sink filter *filter.Filter // Bounded queue — this IS the backpressure mechanism. // Size in BYTES not event count (SEC-12: wide rows must not bypass the limit). queue chan *wal.ChangeEvent // buffered, capacity = config.QueueSize metrics *metrics.Registry } // Called by the WAL stream loop. BLOCKS when queue is full — that's the point. func (d *Dispatcher) Enqueue(ctx context.Context, event *wal.ChangeEvent) error { if !d.filter.Match(event) { return nil } select { case d.queue <- event: d.metrics.QueueDepth.Set(float64(len(d.queue))) return nil case <-ctx.Done(): return ctx.Err() // NO default: case — intentional. Blocking here pauses WAL consumption. // Postgres waits. Memory stays flat. No OOM. } } // Stream loop — note: if Enqueue blocks, ReceiveMessage is not called again. func (s *Stream) run(ctx context.Context) error { for { msg, err := pglogrepl.ReceiveMessage(ctx, s.conn) // ... decode ... if err := s.dispatcher.Enqueue(ctx, event); err != nil { return err // context cancelled = clean shutdown } // Next ReceiveMessage only called after Enqueue returns. } }
Dispatcher Architecture
type Sink interface { Name() string Send(ctx context.Context, batch []*wal.ChangeEvent) error Close() error } // DispatchAndWait fans out to all sinks concurrently, blocks until all confirm. // LSN is only advanced after this returns nil. func (d *Dispatcher) DispatchAndWait(batch []*wal.ChangeEvent) error { var wg sync.WaitGroup errs := make([]error, len(d.sinks)) for i, sink := range d.sinks { wg.Add(1) go func(i int, s Sink) { defer wg.Done() errs[i] = s.Send(context.Background(), batch) }(i, sink) } wg.Wait() return errors.Join(errs...) }
PATCH 6: SSRF & DNS Protection — No External Exploitation

Walveil makes outbound HTTP calls to URLs that users configure. A misconfigured or malicious URL can point to internal services, other containers in your network, or cloud metadata endpoints like 169.254.169.254 which on AWS and GCP serve IAM credentials.

The defense is to resolve the hostname yourself before opening the connection, then inspect every resolved IP address against a blocklist. Do this check on every request attempt, not just at startup — DNS records can change between retries and an attacker can exploit that window.

// internal/delivery/webhook/validate.go var blockedCIDRs = []*net.IPNet{ mustParseCIDR("10.0.0.0/8"), // RFC 1918 mustParseCIDR("172.16.0.0/12"), // RFC 1918 mustParseCIDR("192.168.0.0/16"), // RFC 1918 mustParseCIDR("127.0.0.0/8"), // loopback mustParseCIDR("169.254.0.0/16"), // link-local / metadata endpoints mustParseCIDR("::1/128"), // IPv6 loopback mustParseCIDR("fc00::/7"), // IPv6 private mustParseCIDR("fe80::/10"), // IPv6 link-local } func ValidateAndResolve(rawURL string) (string, error) { u, err := url.Parse(rawURL) if err != nil { return "", err } if u.Scheme != "https" { return "", fmt.Errorf("only https allowed") } addrs, err := net.LookupHost(u.Hostname()) if err != nil { return "", err } for _, addr := range addrs { ip := net.ParseIP(addr) if ip == nil || ip.IsUnspecified() { return "", fmt.Errorf("blocked: unspecified address") } for _, cidr := range blockedCIDRs { if cidr.Contains(ip) { return "", fmt.Errorf("blocked: %s resolves to private/reserved address %s", u.Hostname(), ip) } } } return rawURL, nil } // Custom dialer that re-validates on EVERY connection attempt (DNS rebinding defense) func SafeDialer() *net.Dialer { return &net.Dialer{ Control: func(network, address string, c syscall.RawConn) error { host, _, _ := net.SplitHostPort(address) _, err := ValidateAndResolve("https://" + host) return err }, } }

PHASE 3
Observability & Slot Warden
You cannot operate what you cannot see. Metrics, structured logging, health endpoints, and the slot warden goroutine that protects your Postgres instance.
PATCH 3: Slot Control — No Database Death

A replication slot forces Postgres to retain all WAL from your last acknowledged LSN forward, indefinitely. If walveil stalls for any reason — a slow sink, a network partition, a bad deploy — that WAL accumulates on disk with nothing stopping it. On a busy database this can fill the disk in hours and crash the entire Postgres instance, affecting every application using that database, not just walveil.

You need a warden goroutine that continuously monitors the gap between your confirmed LSN and the current WAL position. It should always alert when lag crosses your threshold. It should only drop the slot if you've explicitly configured it to, because dropping the slot is a destructive act that requires a recovery plan.

// internal/warden/warden.go type Warden struct { db *pgconn.PgConn slotName string maxLagBytes int64 maxLagSecs int64 dropOnLimit bool // explicit opt-in only — defaults to false metrics *metrics.Registry log *zap.Logger } func (w *Warden) Run(ctx context.Context) { ticker := time.NewTicker(30 * time.Second) defer ticker.Stop() for { select { case <-ticker.C: w.check(ctx) case <-ctx.Done(): return } } } func (w *Warden) check(ctx context.Context) { const q = ` SELECT pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn) AS lag_bytes, EXTRACT(EPOCH FROM (now() - restart_lsn_time)) AS lag_secs FROM pg_replication_slots WHERE slot_name = $1` var lagBytes, lagSecs int64 // ... query execution ... w.metrics.SlotLagBytes.Set(float64(lagBytes)) w.metrics.SlotLagSeconds.Set(float64(lagSecs)) over := lagBytes > w.maxLagBytes || lagSecs > w.maxLagSecs if over { w.log.Error("CRITICAL: replication slot lag exceeds threshold", zap.Int64("lag_bytes", lagBytes), zap.Int64("lag_secs", lagSecs), zap.String("slot", w.slotName), ) w.metrics.SlotLagAlerts.Inc() if w.dropOnLimit { w.log.Error("dropping replication slot to protect database") w.dropSlot(ctx) // destructive — requires explicit config opt-in } } }

PHASE 4
Security Hardening
Tighten every security control. TLS enforcement, secret resolution, privilege checks, and full column-level redaction.
TLS Enforcement (SEC-04)

Default sslmode=verify-full on the replication connection. Reject downgrades unless allow_insecure_tls = true is explicitly set in config with a loud startup warning. All webhook HTTP clients must use a tls.Config that enforces MinVersion tls.VersionTLS12.

func buildHTTPClient(cfg WebhookTarget) *http.Client { tlsCfg := &tls.Config{MinVersion: tls.VersionTLS12} transport := &http.Transport{ DialContext: SafeDialer().DialContext, // SSRF guard on every dial TLSClientConfig: tlsCfg, } return &http.Client{ Transport: transport, Timeout: time.Duration(cfg.TimeoutMs) * time.Millisecond, } }
Privilege Check at Startup (SEC-09)

After connecting, verify the replication user is not a superuser. Log a loud warning and optionally fail if it is.

const q = `SELECT rolsuper FROM pg_roles WHERE rolname = current_user` var isSuperuser bool // ... query ... if isSuperuser { log.Warn("⚠ replication user has superuser privileges — principle of least privilege violated (SEC-09)") if cfg.FailOnSuperuser { return fmt.Errorf("refusing to run as superuser") } }
Column-Level Redaction (SEC-06)

The redact_columns config list is applied at the decoder level — before events reach any sink or logger. Format: schema.table.column or *.*.column for global redaction.

func applyRedaction(event *wal.ChangeEvent, rules []string) { for _, col := range rules { parts := strings.Split(col, ".") table, field := parts[0], parts[1] if table == "*" || table == event.Table { if event.After != nil { event.After[field] = "[REDACTED]" } if event.Before != nil { event.Before[field] = "[REDACTED]" } } } }

Graceful Shutdown Sequence
01
Signal received (SIGTERM / SIGINT)
Cancel the root context. Stop accepting new WAL messages.
02
Drain dispatcher queue
Wait for all in-flight events to be delivered and acknowledged. Bounded by ShutdownTimeout (default 30s).
03
Flush checkpoint
Write the final confirmed LSN to the checkpoint file. This is the critical last step — do it before closing connections.
04
Send final StandbyStatusUpdate
Acknowledge the final LSN to Postgres before closing the replication connection.
05
Close all connections and exit 0
Close replication conn, Redis client, HTTP client. Exit cleanly.

APPENDIX B
PostgreSQL Prerequisite Checklist
Everything the database needs before walveil can connect.
  • wal_level = logical in postgresql.conf — requires restart. Verify with SHOW wal_level at startup.
  • Publication exists: CREATE PUBLICATION walveil_pub FOR TABLE public.orders, public.users;
  • Replication user created with REPLICATION privilege only — NOT superuser (SEC-09)
  • Tables requiring old row values in UPDATE/DELETE have REPLICA IDENTITY FULL or REPLICA IDENTITY USING INDEX
  • Replication user has SELECT on all tables in the publication
  • TLS configured on Postgres if using sslmode=verify-full (SEC-04)
  • max_replication_slots has capacity for at least one additional slot
  • max_wal_senders has capacity for the replication connection
